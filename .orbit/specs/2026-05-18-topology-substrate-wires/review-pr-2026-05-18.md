## Findings

### [MEDIUM] MCP parity tests not added for the three new envelope-extending verbs
**Category:** test-gap
**Description:** ac-02, ac-03, and ac-04 verification statements each explicitly require *"Parity test on CLI + MCP"* (ac-02: "Parity test in orbit-state/crates/cli/tests/parity.rs and orbit-state/crates/mcp/tests/parity.rs covers all four cases"; ac-03: "Parity test on CLI + MCP"; ac-04: "Parity test on CLI + MCP confirms the envelope shape"). The implementation adds CLI parity tests in `crates/cli/tests/parity.rs` but no corresponding MCP tests in `crates/mcp/tests/parity.rs`. `crates/mcp/tests/common/mod.rs` was touched to add `topology_warnings: vec![]` to the two existing canonical envelopes — sufficient to keep the existing MCP spec.close tests green with the new field, but it does not exercise the populated-drift / populated-warnings / populated-nudge code paths over the MCP transport.
**Evidence:**
- `grep -nE "fn .*topology|nudge|topology_drift|topology_warnings" orbit-state/crates/mcp/tests/parity.rs` → no matches.
- `grep -n "topology" orbit-state/crates/mcp/tests/common/mod.rs` → only the two `topology_warnings: vec![]` lines (397, 437) on the existing spec_close envelopes.
- Compare to CLI: `crates/cli/tests/parity.rs` adds 9 new tests (lines 952-1255) over the same three ACs.
**Recommendation:** Add MCP parity tests mirroring the CLI ones, at minimum:
  1. `session_prime_mcp_topology_drift_omitted_when_not_configured` (collapsed: both file-absent and docs.topology-unset).
  2. `session_prime_mcp_topology_drift_populated_when_drift_present`.
  3. `spec_close_mcp_topology_warnings_populated_on_word_boundary_match`.
  4. `memory_remember_mcp_topology_label_envelope_carries_nudge`.
  5. `memory_remember_mcp_no_nudge_flag_suppresses_envelope_nudge`.
  These can ride alongside the existing MCP envelope parity pattern at `crates/mcp/tests/parity.rs:254-311` (audit_drift / overview / card_specs). The CLI tests already validate the JSON envelope shape; the MCP tests would lock the same shape over the MCP transport — the contract the verification language pins down.

  This is MEDIUM rather than CRITICAL because the core-level unit tests directly exercise the underlying logic on `SpecCloseResult`, `SessionPrimeResult`, and `MemoryRememberResult`; the MCP transport is a thin wrapper over these (verified by the existing canonical-envelope parity tests for the unchanged shape). Risk surface is regression in MCP serialisation specifically, not in the topology logic itself.

The implementation is precise. The MEDIUM finding is the load-bearing one: the spec is explicit about CLI + MCP parity coverage for the three new envelope-extending ACs, and only the CLI side shipped. The core logic is exercised by 13 unit tests on `core/src/verbs.rs` and the CLI envelope shape is exercised by 9 parity tests on `cli/tests/parity.rs`, so the risk is narrowly the MCP transport layer's serialisation of these three new fields. That layer is a thin wrapper over the core verbs and the existing MCP canonical-envelope parity tests (`audit_drift_mcp_envelope_matches_canonical_envelope`, etc.) demonstrate the pattern. The gap is real but the blast radius is contained — hence APPROVE with the recommendation to add MCP parity in a follow-up commit, not BLOCK.

ac-01's bash-script test gap is the LOW; manual smoke confirms the six fixture cases the verification names, but they aren't codified. Reasonable as-is for a small bash script with high prose-spec coverage; codify if regression pressure appears.

Net: ship. The MEDIUM is worth a small follow-up commit before this work goes far from memory, but it isn't a merge blocker — the underlying behaviour is verified at both the unit and CLI-transport levels.
