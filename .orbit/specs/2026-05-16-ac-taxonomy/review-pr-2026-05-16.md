## Findings

### [LOW] `let _ = ensure_current(layout)` swallows migration errors that round-trip cannot detect
**Category:** edge-case
**Description:** `verify.rs:92` discards the `Result` from `ensure_current(layout)` with the comment claiming "errors here surface as round-trip failures on the schema-version path so the caller sees a single channel of diagnostics." This is true for some failure modes but **not** for the realistic case of a future-versioned schema-version file. Concretely: a user on an older binary scanning a newer tree (e.g. on-disk `version: '0.99'`, binary at `CURRENT_SCHEMA_VERSION = "0.3"`) will hit `migration.run` returning `malformed: schema-version file has unknown version`. The discard at line 92 swallows that error. The subsequent `check_round_trip::<SchemaVersion>` at line 98 reads the file, parses it cleanly, re-serialises identical bytes, and reports OK. `verify_all` returns `VerifyOutcome::default()` — clean. The user gets a green-light verify against a tree the binary fundamentally cannot understand.
**Evidence:** `verify.rs:92` `let _ = ensure_current(layout);`. `migrations.rs:234-243` returns `malformed` on unknown version. `verify.rs:158-198` `check_round_trip` only fails on read/parse/serialise/byte-diff — not on "version this binary doesn't recognise." No test exercises this path: the existing `verify_repairs_known_schema_version_drift_via_migration` test covers the *known-older-version* case (which works correctly because the migration runs).
**Recommendation:** Either (a) record a `MigrationFailure` variant on `VerifyOutcome` and surface ensure_current errors there so verify reports the failure; or (b) match on the error category and only swallow the `malformed`/`not_found` cases that round-trip *will* re-detect, propagating others. Lowest-cost fix: add a separate `outcome.migration_errors: Vec<String>` channel and push to it on Err, factoring `has_failures()` to include it. Not a merge-blocker — the spec doesn't ship forward-incompat detection as an AC, the bug only manifests on rollback or skewed-binary scenarios which are already covered by the CHANGELOG forward-incompat note. Worth a memo as a follow-up.

### [INFO] `transform_detail` field forward-compat posture — sound for the consumer pattern
**Category:** edge-case
**Description:** `DispositionRecord` (`reconcile.rs:298`) gains an `Option<String> transform_detail` field with no `skip_serializing_if`, so every record serialised by 0.4.16+ carries `transform_detail: ~` (or null) in JSON. The struct itself does not derive `deny_unknown_fields`, and DispositionRecord is an OUTPUT of the reconcile verb (consumed by `canonicalise --reconcile` callers from JSON envelopes) rather than an INPUT serde-parsed back into the struct. So the additive field cannot break a 0.4.15 binary — there is no 0.4.15 code path that re-parses a DispositionRecord. The brief's stress-test concern checks out: no caller chokes on the addition. This is the correct pattern for forward-additive output structs.
**Evidence:** `reconcile.rs:298-316`. `grep -rEn "DispositionRecord" orbit-state/` shows callers only construct or read fields — none re-parse. The CLI renderer at `cli/src/main.rs` reads via field access only.
**Recommendation:** None. Note here for future maintainers — when this field becomes load-bearing for a downstream consumer, consider adding `#[serde(default, skip_serializing_if = "Option::is_none")]` for tidier JSON output, but it's stylistic until then.

## Honest Assessment

The implementation is exceptionally clean for a 15-AC spec that touched 35 files across the Rust core, both CLI/MCP parity surfaces, three SKILL.md files, the substrate METHOD.md, and a brewed-binary release. Test coverage is dense (40+ new tests, every AC carrying a `// spec 2026-05-16-ac-taxonomy ac-NN` traceability comment so the future reviewer can find the contract in the test source), the schema migration is idempotent and partially-recoverable per the runner's existing contract, and the `Disposition::Transform` variant is purely additive — every existing `Map / Drop / Quarantine` caller continues to work and is exercised by regression tests. The `time_gated_open → deferrable_open` rename is complete: a fresh `grep -rEn "time_gated_open|time_gated" orbit-state/ plugins/` returns only intentional prose mentions in archived sidecar review docs (`.orbit/specs/2026-05-13-spec-close-ac-preflight/...` and the cycle-1/v2 spec reviews of this very spec), with zero hits in live code, skills, or scripts. The CHANGELOG honestly names the forward-incompatibility (a 0.4.15 binary cannot read a 0.4.16 spec.yaml carrying `ac_type` because of `deny_unknown_fields`), the brownfield-reconcile gap surfaced by ac-12 is captured in a thorough memo with a clear next-step pointer rather than buried, and the brewed-binary smoke (ac-14) confirmed the brew tap update propagated cleanly. The one non-trivial finding — `let _ = ensure_current(...)` swallowing migration errors for unknown future versions — is a low-severity edge case that doesn't block this merge but is worth a follow-up memo. APPROVE on the strength of the engineering and the discipline; the deferred ac-12 is the right call given the scope of the brownfield drift it surfaced.
